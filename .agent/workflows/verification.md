# 科學驗證協議 (Test-Driven Fix Verification)

1. **嚴禁盲猜與白老鼠測試**
   * 當第一次修改後問題依舊存在，絕對 **不可以** 再次修改後直接說「請您再試試看」。必須寫腳本證明可以運作。

2. **撰寫自動化測試腳本 (E2E / Integration)**
   * 在 `scripts/` 目錄下建立針對該問題的專用測試腳本，例如 `scripts/test_ui_interactive.js`。
   * 如為 UI 互動問題，應使用 `Playwright` 啟動無頭瀏覽器模擬真實操作 (如點擊按鈕、等待 Modal 出現等)。
   * 如為後端 API 問題，應撰寫 fetch 腳本對本地端 port 進行直接存取與 Assert 驗證。

3. **執行驗證**
   * 運行 `node scripts/test_xxx.js`（請注意 `package.json` 若為 `type: module`，請使用 ES6 語法）。
   * 等待測試腳本顯示 **PASS**。

4. **確認修復後回報**
   * 只有在「腳本親自跑過，並且毫無報錯、完全符合預期狀態」後，才能將結果並附上「測試腳本的執行紀錄」回報給使用者。
   * 若測試有錯誤，視為尚未完成，繼續修正直到綠燈 (Green Light) 為止！
